# 全局规范

## 1. 文档管理

* **命名格式**：`文件名_YYYYMMDD_v主.次.md`
* **版本演进**：主版本为整数，次版本为字母（例：`0.a` -> `0.b` -> `1.0` -> `1.a` -> `2.0`）。
* **修订记录**：所有文档**必须**包含修订记录表格。
* **修改逻辑**：默认在当前文档修改并更新版本号，除非明确要求创建新版本。

## 2. 代码控制

* **推送权限**：**严禁**主动执行 `git push`。任何推送到远端（如 GitHub）的操作前，**必须**获取用户的明确授权。即使任务完成也不得例外。

## 3. 解决问题思路

* **需求评审**：认真评审需求，敢于质疑。
* **执行计划**：先计划后执行，重大操作先询问。
* **敏捷迭代**：兼顾简洁与性能；冲突时，先求功能实现（低性能版），后续迭代再追求高性能。
* **环境适配**：方案需兼顾 Windows 笔记本 (ThinkPad X1) 与 Mac mini M4 (2024) 双平台。

## 4. 技能与文档语言规范

### 4.1 技能提示词 (Skill Prompts)
* **执行版本**：提示词主体使用英文编写（AI执行效率更高）
* **中文对照**：必须为每个提示词提供中文版本，命名格式为 `{prompt-name}-zh.md`，用于解释说明，不用于执行
* **示例**：`project-planner-prompt.md` (执行用) + `project-planner-prompt-zh.md` (解释用)

### 4.2 技能输出文档
* **语言要求**：所有项目管理相关文档必须以中文为主
* **适用范围**：包括但不限于 `0-initial-req.md`、`0.5-high-level-arch.md`、`1-project-plan.md`、`2.1-iter-plan.md`、`2.2-algo-design.md`、`2.3-acceptance.md`
* **目的**：便于团队成员（非技术背景）查看和理解

### 4.3 代码与注释
* **代码本身**：保持英文（变量名、函数名、类名）
* **注释**：关键逻辑使用中文注释，便于团队维护
